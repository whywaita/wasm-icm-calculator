# Decisions: WASM ICM Calculator

## Decision Records

| # | カテゴリ | 質問 | 決定事項 | 根拠 |
|---|---------|------|---------|------|
| 1 | 矛盾・曖昧さ | `PkoConfig` が `CalculationInput` から参照されていない | `CalculationInput` に `pkoConfig?: PkoConfig` フィールドを追加する | 型安全性を確保し、PKO 固有の設定を明示的に渡す |
| 2 | 矛盾・曖昧さ | Standard Bounty の `BountyChip_value(i)` の具体的な計算ロジックが未定義 | **ノックアウト獲得モデル**: 各プレイヤーの `bountyChips` は「そのプレイヤーをノックアウトしたらもらえる金額」。自分の bounty は自分がノックアウトされたときに相手に渡る。期待値はノックアウト確率（スタック比例）から計算 | 当初の「チップ比例でプール分配」から変更。実際のトーナメントメカニクスに合致 |
| 3 | 矛盾・曖昧さ | Breakeven の `buyIn` は「レーキ含む」だが `prize_return_rate` との関係が曖昧 | **Entry fee UI**: ユーザーは entry_fee（総額）を入力 → UI が自動で buy-in と rake に分解（デフォルト rake 10%）→ 手動変更も可能。`effective_buyin = entry_fee - rake` | ユーザーが支払う総額を入力するのが直感的 |
| 4 | 計算量 | 厳密計算 n<=20 で O(n!/(n-p)!) が sub-second で収まるか | **メモ化で対応**。再帰計算の部分結果をキャッシュし計算量を大幅に削減する | ブラウザ環境のメモリ制約内でメモ化は十分実現可能 |
| 5 | アーキテクチャ | WASM 計算でメインスレッドがブロックされる | **Web Worker で WASM を実行**する。UI スレッドとは別スレッドで計算を行う | 計算に数百 ms かかる可能性があり、UI フリーズを防ぐ |
| 6 | エッジケース | 入力バリデーション（スタック 0/負、プライズ合計超過等）をどこで行うか | **WASM 側（Rust）でバリデーション**。エラーメッセージを JSON で返す | バリデーションロジックの一元管理。Rust の型システムで安全性を確保 |
| 7 | API 設計 | 4つの WASM 関数に分かれているが breakeven は内部で ICM を呼ぶのか | **単一の `calculate` 関数に統合**。`tournamentType` で内部分岐し、breakeven も含めて一括計算 | API がシンプルになり、JS 側の呼び出し管理が不要になる |
| 8 | PKO 再帰 | PKO バウンティ期待値の再帰計算の終了条件 | **動的カットオフ**: `r^depth < 0.1`（r = 継承率）で打ち切り | 固定3人では高継承率（r=0.8）で残存 51.2% が無視される。動的カットオフで残存を常に 10% 以下に保証 |
| 9 | Bounty 単位 | `bountyChips` の単位はドルかチップか | **単位なしの数値**（表記は ICM$ を使用）。ノックアウト時に獲得する金額を表す | 通貨単位は不要。ICM$ として統一的に扱う |
| 10 | テスト戦略 | ICM 計算の正確性検証方法 | **プロパティベースドテスト中心**。「全エクイティ合計 = プライズプール」「チップリーダーが最大エクイティ」等の不変条件で検証 | バウンティ/PKO に参照実装がないため、具体値の照合よりも不変条件の検証が有効 |
| 11 | ライセンス | 具体的なライセンスの選択 | **Apache 2.0** | 特許保護付きの permissive ライセンス |
| 12 | WASM サイズ | 初回ロード時の UX | **特に対策不要**。Rust + wasm-pack は十分コンパクトなバイナリを生成する想定 | 実測で問題があれば後から wasm-opt を追加 |

## Open Questions (Resolved)

All previously open questions have been resolved:

- [x] 近似アルゴリズム → **モンテカルロ**（ランダム脱落順序サンプリング）
- [x] 最大サポートプレイヤー数 → **50人**
- [x] PKO ノックアウト確率モデル → **スタック比例のみ**（v1）
- [x] 入力プリセット → **プレイヤースタックプリセットは v1 ではなし**、ペイアウト構造プリセットは提供
- [x] 結果の可視化 → **テーブル + エクイティ棒グラフ + ICM プレッシャーカーブ**
- [x] i18n → **英語（デフォルト）+ 日本語を v1 から**
- [x] Standard Bounty 確率モデル → **ペアワイズスタック比例**（`stack_i / (stack_i + stack_j)`）
- [x] PKO 打ち切り深さ → **動的: `r^depth < 0.1`**（継承率に応じてカットオフ深度が変化）
- [x] メモ化キャッシュ戦略 → **`u32` ビットマスク + `HashMap<(usize, u32), f64>`、payout positions で枝刈り**
